# My Heart / Mochi

## 概述

My Heart（Mochi）是一款端侧语音到动作的闭环具身交互产品。它将语义化的语音指令桥接到物理硬件执行，无需依赖云端，通过一条完整的处理链路展现了软硬件一体化能力：语音活动检测、自动语音识别、大语言模型推理、手势关键词路由，以及通过 UART 串口协议实现的双自由度舵机控制。

系统完全在设备端运行，通过本地 Whisper ASR 处理语音输入，使用滑动的 10 轮窗口维护对话上下文，将指令路由为物理动作，并通过 Edge TTS 提供音频反馈。这套架构消除了云端延迟，能够在离线环境中实现可靠的实时交互。

## 端到端状态机

交互链路由一个具有五种状态的确定性状态机管理：

- **IDLE** — 等待语音活动
- **LISTENING** — 通过麦克风 VAD 采集音频输入
- **PROCESSING** — 运行 ASR → 上下文组装 → LLM 推理 → 手势路由
- **SPEAKING** — 播放 TTS 反馈并执行物理指令
- **ERROR** — 处理超时、硬件故障或无法识别的输入

正常流程：**IDLE → LISTENING → PROCESSING → SPEAKING → IDLE**。ERROR 状态可从任意阶段分支进入，在记录失败并播放错误提示后返回 IDLE。

该状态机确保同一时刻只有一个阶段在执行，避免了音频采集与播放之间的竞态条件，并保证在麦克风处于活动状态时绝不会下发舵机指令。

## 八节点数据流

完整的语音到动作链路由八个顺序处理节点组成：

### 1. 麦克风 VAD（语音活动检测）

音频从麦克风采集，并使用 **RMS 能量**阈值结合**静默计数**进行实时分析。当 RMS 能量在最短持续时间内超过配置的阈值时，系统从 IDLE 转入 LISTENING。当能量低于阈值并达到静默计数上限时，音频片段被最终确定并传递给 ASR。

这种基于 VAD 的自动分段消除了手动按键说话（push-to-talk）触发的需要，实现了自然的对话流。

### 2. 本地 Whisper ASR（自动语音识别）

采集到的音频片段使用在设备本地运行的 **Whisper** 进行转写。Whisper 模型在启动时加载一次，并在 CPU 上执行推理，返回用户语音的文本转录。

本地 ASR 确保零网络依赖，并保证语音数据绝不离开设备，从而解决了基于云端的语音服务固有的隐私顾虑。

### 3. 滑动窗口上下文（10 轮历史）

当前转录结果被追加到**最近 10 轮的滑动窗口**中（5 条用户话语 + 5 条助手回复）。该上下文以简单列表的形式保存在内存中，并作为对话历史传递给 LLM。

当上下文超过 10 轮时，最旧的一轮被丢弃（FIFO）。这种有界内存机制在保留足够近期历史以支撑连贯的多轮对话的同时，防止了上下文的无限增长。

### 4. OpenAI 兼容 LLM（通过 YAML 可切换）

组装好的上下文被发送到一个 **OpenAI 兼容的大语言模型**端点。LLM 服务通过**抽象工厂模式**封装在一个接口之后：具体实现（OpenAI API、本地 Ollama，或任意 OpenAI 兼容的提供方）由单行 YAML 配置指定。

这种解耦使得无需修改代码即可更换供应商。例如，从 OpenAI GPT-4 切换到本地 Llama 模型，只需更新 YAML 中的 `llm.provider` 和 `llm.endpoint` 字段。

LLM 返回一段自然语言回复，并可选地在输出中嵌入一条结构化的手势指令。

### 5. 手势关键词路由

系统解析 LLM 回复中的**手势关键词**（例如 "look_left"、"nod"、"look_up"）。一个简单的关键词匹配器提取这些指令，并将其映射为物理舵机位置。

例如：
- `"look_left"` → H30（水平舵机转到 30 度）
- `"look_right"` → H150（水平舵机转到 150 度）
- `"nod"` → V60（垂直舵机转到 60 度）

如果未找到手势关键词，系统跳过物理执行，直接进入 TTS 反馈。

### 6. UART 驱动（9600 波特，ASCII 协议）

检测到的手势被编码为 **ASCII UART 指令**，并以 **9600 波特**通过串口传输。每条指令遵循一个简单的文本协议：

```
H90,V120\n
```

其中：
- `H` = 水平舵机角度（0–180）
- `V` = 垂直舵机角度（0–180）
- `\n` = 换行终止符

UART 驱动强制执行**位置限制**（0–180 度）以防止硬件损坏。指令通过 `pyserial` 同步发送，并设有短超时以检测通信故障。

### 7. Arduino 固件（逐度 15ms 插值）

Arduino Nano ATmega328P 接收 UART 指令，解析 `HX,VY` 格式，并执行**逐度 15 毫秒步进插值**来平滑舵机运动。

固件不会命令舵机直接跳到目标角度（这会导致可见的抖动和机械应力），而是以 15ms 的步间延迟，每次将舵机位置递增一度。这种固件级插值抑制了物理抖动，产生平滑、自然的运动。

固件在一个紧凑循环中运行：读取 UART → 解析指令 → 插值到目标 → 重复。如果在超时周期内没有收到指令，舵机保持当前位置。

### 8. 双自由度 SG90 舵机控制 + Edge TTS 反馈

两个 **SG90 舵机**提供水平和垂直方向的自由度。舵机通过 Arduino 生成的 PWM 信号控制，位置反馈由舵机内部电位器提供（在当前实现中被读取，但未主动用于闭环控制）。

与此同时，LLM 的文本回复使用 **Edge TTS** 合成为语音，并通过 `PyAudio` 经设备扬声器播放。这段音频反馈闭合了交互回路：用户在听到助手回复的同时观察到相应的物理手势。

**当前局限：** TTS 播放与舵机运动是**串行**的，而非时间对齐的。系统先播放 TTS，再执行手势。真正的时间对齐动作+语音（例如一边说"是"一边点头）是一个路线图项。

## 抽象工厂服务解耦

系统架构使用**抽象工厂模式**对异构服务（ASR、LLM、TTS）进行解耦。每个服务都通过一个稳定的 Python 接口访问，具体实现在运行时根据 YAML 配置选择。

**服务接口：**
- `ASRService` → `transcribe(audio: bytes) -> str`
- `LLMService` → `generate(context: List[Message]) -> str`
- `TTSService` → `synthesize(text: str) -> bytes`

**配置切换示例：**

```yaml
# Config A: OpenAI cloud services
llm:
  provider: openai
  model: gpt-4
  endpoint: https://api.openai.com/v1

# Config B: Local Ollama + Whisper
llm:
  provider: ollama
  model: llama3
  endpoint: http://localhost:11434/v1
```

修改 YAML 文件并重启服务即可切换整个后端。这种设计实现了：
- **供应商独立性：** 无需修改代码即可从 OpenAI 切换到 Anthropic、本地模型或自定义端点
- **环境可移植性：** 在开发中使用云端 API，在生产/离线部署中使用本地模型
- **A/B 测试：** 仅通过配置即可对比不同 LLM 提供方

## 物理抖动抑制

物理抖动（舵机运动过程中可见的晃动或顿挫）通过两种机制加以抑制：

1. **固件级插值：** 固件不发送单条"移动到 120°"指令，而是逐步递增位置（每 15ms 移动 1°），产生平滑的运动曲线。这将运动平滑的职责下推到硬件层，在那里可以实现严格的时序控制。

2. **基于 VAD 的自动分段：** 通过使用 RMS 能量检测对语音进行分段，系统避免了固定长度录音窗口所导致的突兀起止转换。VAD 能够自然地捕获完整话语，降低了指令被截断的可能性，从而减少令人困惑或不完整的手势。

这些技术以一定的延迟（15ms × N 度）换取显著提升的运动质量。在用户测试中，插值运动被感知为"自然"，而直接位置指令则被描述为"生硬"或"机械"。

## 技术栈

- **Python asyncio** — 用于并发音频 I/O 与服务调用的事件循环
- **Whisper** — 本地自动语音识别
- **OpenAI 兼容 SDK** — 面向 LLM 服务的统一客户端
- **Edge TTS** — 微软 Edge 文本转语音合成
- **PyAudio** — 音频采集与播放
- **NumPy** — 用于 VAD 的 RMS 能量计算
- **Arduino Nano ATmega328P** — 用于舵机控制的微控制器
- **SG90 servos** — 用于双自由度运动的入门级舵机
- **pyserial** — UART 通信库
- **UART 9600 baud** — Python ↔ Arduino 通信的串口协议

## 诚实路线图

当前实现有两个已知局限：

1. **TTS 播放仍是占位方案。** 尽管 Edge TTS 可以工作，但其音质和延迟尚未达到生产可用水平。未来版本将集成更低延迟、更高质量的 TTS 引擎，或预先合成常用回复。

2. **动作与语音是串行的，而非时间对齐的。** 系统先播放 TTS 音频，等待其完成，再执行舵机手势。真正的多模态交互需要时间对齐的执行：例如一边说"是"一边点头。这需要重构状态机，以支持带同步时序的 TTS 与舵机并发执行。

这些局限并不妨碍当前用例（博物馆导览演示、儿童陪伴原型），但在养老照护或商业场景的生产部署中必须加以解决。

## 价值主张

My Heart 将屏幕上的 AI 带入物理世界。它证明了对话式 AI 能够通过一套稳健、解耦的架构控制真实硬件，覆盖从语音输入到物理输出的完整链路。

软硬件解耦（服务层的抽象工厂、Arduino 通信的 UART 协议）使得**仅通过更换提示词和关键词即可移植到新用例：**

- **博物馆导览：** 载入领域专属提示词（艺术史、展品说明）和手势关键词（指向左侧、指向右侧、抬头看）
- **儿童陪伴：** 载入活泼的提示词（讲故事、教育问答）和富有表现力的手势（开心挥手、难过点头）
- **养老照护：** 载入耐心、有同理心的提示词（用药提醒、跌倒检测警报）和辅助手势（招手示意、指向门口）

核心链路（VAD → ASR → LLM → 关键词路由 → UART → 舵机）保持不变。仅需更换 YAML 配置、提示词模板和关键词字典。这种**提示词驱动的可适配性**是关键的产品洞察：单一的具身交互平台通过配置而非重新实现，就能演化为一系列领域专属产品。

## 真实指标

- **UART 波特率：** 9600 波特
- **双自由度分辨率：** H90, V120（水平 0–180°，垂直 0–180°）
- **舵机插值步进：** 每度 15ms
- **上下文窗口：** 10 轮（滑动 FIFO）
- **VAD 静默阈值：** 可配置，通常为 300ms
- **RMS 能量阈值：** 可配置，通常为 0.02 归一化幅值
