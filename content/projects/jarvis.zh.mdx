# 架构概览

Jarvis 是一个多智能体协调中枢，它将大语言模型的语义理解与复杂任务执行解耦。Jarvis 不把任务逻辑直接嵌入 LLM 提示词或 API 调用中，而是提供一个统一的协调层，将对话意图路由到专门的领域智能体，从而把对话式 AI 转化为可靠、可追踪、可溯源的任务执行。

## 设计理念

其核心架构洞见是**两阶段意图识别**：先运行关键词匹配，只有当置信度低于 0.8 时，系统才回退到大语言模型。这种混合方式兼顾速度与语义灵活性——简单且高频的指令通过模式匹配即时解析，而模糊或新颖的请求则借助模型的推理能力，无需在每一次交互中都付出延迟与成本代价。

## 语义缓存

为进一步降低延迟与成本，Jarvis 在 **Redis 中实现了语义缓存**，使用由归一化后的用户输入派生出的 md5 键。每个缓存条目的存活时间为 3600 秒。当语义相同的请求在 TTL 窗口内到达时，系统会取出缓存的意图并完全跳过 LLM 调用，将重复查询的 P95 响应时间缩短一半以上。

## 异步任务引擎

执行层是一个 **asyncio 优先级队列**，包含四个优先级：CRITICAL、HIGH、NORMAL 和 LOW。十个工作协程从队列中拉取任务，并通过信号量将并发操作限制为 100，以防止资源耗尽。系统采用双层超时防护：任务封套层面的 60 秒全局超时，以及各智能体在其注册契约中声明的单智能体超时。

这种架构确保高优先级任务（面向用户的响应、时效性告警）能够抢占批处理或后台工作，且任何单个行为异常的智能体都无法阻塞队列。

## 智能体抽象与注册表

全部 18 个领域智能体都继承自**统一的 BaseAgent 抽象**，它定义了每个智能体必须满足的契约：唯一标识符、一组能力关键词、一个 execute 方法，以及声明的超时与重试策略。中心化的 **AgentRegistry** 管理智能体的生命周期：注册、能力查找、运行时热重载（无需重启协调器即可更新或新增智能体），以及自动回退（若某个智能体健康检查失败，注册表会将请求路由到回退智能体或返回优雅的错误）。

这种抽象消除了脆弱的、针对特定智能体的分发逻辑，使 Jarvis 成为一个可扩展的能力中枢，而非硬编码的任务编排器。

## 重试策略与延迟目标

每个智能体都声明自己的重试策略；默认策略是**三次重试，采用 0 秒、2 秒、4 秒的指数退避**。瞬时性故障（网络超时、限流）由重试层吸收，而持续性故障则以结构化的错误响应向上冒泡。系统的目标是端到端请求处理的 **P95 延迟低于 200 毫秒**，从接收意图到任务分发进行度量。

## 技术栈

该实现基于 **Python 3.11 asyncio** 实现协作式并发，采用 **FastAPI** 作为 HTTP API 层，使用 **带 JSONB 列的 PostgreSQL** 存储结构化的任务与智能体状态，使用 **Redis** 进行语义缓存与进程间信号传递，使用 **Prometheus** 导出指标，并使用 **Docker** 进行容器化部署。异步原生的设计比线程或进程池模型更高效地利用 CPU 与 I/O，而 Prometheus 集成则暴露了延迟直方图、队列深度以及各智能体的成功率，以实现实时可观测性。

## 价值主张

Jarvis 将对话式 AI 转化为可靠、可追踪、可溯源的任务执行。通过将语义理解与任务编排解耦，它提供了一个具备容错调度能力的可扩展能力中枢，使得新增智能体、切换 LLM 提供商，或在任务量增长时横向扩展都变得简单直接。相较于将意图识别与业务逻辑纠缠在一起的单体式智能体框架，该架构更看重可观测性、优雅降级与运维简洁性。
